# CodeLens — Система семантического поиска по исходному коду

Проект представляет собой инструмент интеллектуального поиска по кодовой базе Python-приложения (на примере репозитория `gymhero`). Система принимает поисковые запросы на естественном языке, находит наиболее релевантные классы и функции, а также умеет генерировать краткие технические пояснения с помощью локальной языковой модели.

---

## Структура проекта и назначение файлов

- **`index.py`** — Скрипт для создания поисковой базы данных. Он обходит указанную папку, с помощью синтаксического анализа (AST) разбивает `.py` файлы на логические фрагменты (функции и классы), генерирует для них векторные эмбеддинги и сохраняет всё в базу ChromaDB.
- **`search_engine.py`** — Поисковый движок. Реализует алгоритм гибридного поиска: рассчитывает косинусное сходство между вектором запроса и кодом в базе, а также начисляет текстовые бонусы за точные совпадения ключевых слов в названиях и теле функций.
- **`app.py`** — Графический веб-интерфейс приложения, разработанный на Streamlit. Позволяет удобно вводить запросы в браузере, искать нужные участки кода с подсветкой синтаксиса и получать ответы от ИИ.
- **`llm_assistant.py`** — Модуль интеграции с локальной языковой моделью через Ollama. Он берёт найденные фрагменты кода, формирует промпт и отправляет запрос к модели `mistral:7b` для генерации суммаризации на русском языке.
- **`generate_results.py`** — Скрипт для автоматического тестирования системы по набору вопросов от организаторов (`eval_questions.json`) с сохранением результатов в итоговый файл.
- **`requirements.txt`** — Список всех библиотек и внешних зависимостей, необходимых для работы программы.
- **`eval_questions.json`** — Тестовый набор вопросов для проверки точности поиска.
- **`results.json`** — Файл с результатами работы поискового движка по тестовым вопросам.
- **`score.py`** — Скрипт, предоставленный организаторами, для проведения оценки работы точности поиска (Precision@5).

## Архитектура системы

Система состоит из четырёх основных компонентов, реализованных на Python:

1. **Индексатор (`index.py`)**  
   Обходит указанную директорию, находит все `.py`-файлы, с помощью встроенного модуля `ast` выделяет функции и классы (включая их docstring и тело). Для каждого фрагмента с помощью `sentence-transformers` вычисляется векторное представление (эмбеддинг). Полученные векторы и метаданные (имя, тип, путь к файлу, строки кода) сохраняются в векторную базу данных ChromaDB.

2. **Векторная база данных (ChromaDB)**  
   Хранит эмбеддинги размерностью 384 и поддерживает поиск ближайших соседей по косинусному расстоянию. Используется встроенный режим (без отдельного сервера), что упрощает развёртывание.

3. **Поисковый движок (`search_engine.py`)**  
   Реализует **гибридный поиск**:  
   - векторное сходство между эмбеддингом запроса и эмбеддингами фрагментов;  
   - текстовые бонусы за точное вхождение слов запроса в название функции/класса, docstring и тело кода.  
   Итоговый рейтинг вычисляется как взвешенная сумма этих двух оценок.

4. **Веб-интерфейс (`app.py`)**  
   Построен на Streamlit и предоставляет:  
   - поле ввода запроса на русском или английском;  
   - вывод топ-5 результатов с подсветкой синтаксиса (`st.code`);  
   - отображение оценки релевантности в процентах;  
   - опциональный режим генерации связного ответа через LLM.

5. **LLM-ассистент (`llm_assistant.py`)** – опциональный компонент, интегрирующий локальную модель `mistral:7b` через Ollama. Получает найденные фрагменты и вопрос пользователя, формирует промпт и возвращает краткое техническое объяснение на русском языке.
## Обоснование ключевых решений

### Стратегия чункования (разбиения на фрагменты)

Выбрано разбиение на уровне функций и классов, поскольку:
- Функция/класс является **минимальной семантически завершённой единицей** в Python-коде – она выполняет конкретную задачу и имеет чёткую сигнатуру.
- Такой подход сохраняет контекст внутри фрагмента, что улучшает качество эмбеддингов и релевантность поиска.
- Docstring (если есть) включается в тело фрагмента, что обогащает текстовое описание и помогает при гибридном поиске.

### Выбор модели эмбеддингов

Используется `paraphrase-multilingual-MiniLM-L12-v2` из библиотеки `sentence-transformers`:
- Поддерживает **русский и английский** языки, что критично для запросов на обоих языках.
- Размерность вектора 384 – хороший баланс между скоростью вычислений и качеством представления.
- Лёгкая модель, работает на CPU без существенных задержек.

### Выбор векторной базы данных

Предпочтение отдано **ChromaDB**:
- Встраиваемая (embedded), не требует установки отдельного сервера.
- Проста в использовании, полностью интегрируется с Python.
- Достаточно производительна для базы из нескольких сотен фрагментов.

### Гибридный поиск

Комбинация векторного поиска и текстовых бонусов позволяет:
- Находить семантически близкие фрагменты даже при отсутствии точных совпадений слов.
- Повышать рейтинг фрагментов, где ключевые слова запроса встречаются в названии или docstring – это часто указывает на прямое соответствие.

## Язык запросов

Система поддерживает ввод поисковых запросов **как на русском, так и на английском языке** благодаря использованию мультиязычной модели эмбеддингов. Это позволяет единообразно обрабатывать вопросы, заданные на любом из этих языков.

## Подготовка данных для индексации

Система индексирует Python-файлы (`.py`) из директории, которую вы укажете при запуске `index.py`.
Вы можете использовать:

- **Предоставленный организаторами датасет** — распакуйте архив `codebase_python.zip` в корневую папку проекта. Внутри будет папка (например, `gymhero`), содержащая ~80 Python-файлов.
- **Любой другой Python-проект** — скопируйте его в удобное место и укажите путь к нему при индексации.

**Важно:** 
- Индексатор обрабатывает только файлы с расширением `.py`. 
- Вложенные директории обходятся рекурсивно. 
- Файлы с синтаксическими ошибками пропускаются (выводятся в консоль).

**Пример расположения:**  
Если вы распаковали датасет в папку `./gymhero`, то команда индексации будет:
```bash
python index.py ./gymhero
```

## Инструкция по запуску проекта

Для запуска и проверки проекта последовательно выполните следующие команды в терминале, запущенном в директории проекта.

### 1. Установка зависимостей

**Linux / macOS:**
```bash
python3.12 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

**Windows (командная строка):**

```bash
python -m venv .venv
.venv\Scripts\activate.bat
pip install -r requirements.txt
```
## 2. Активация виртуального окружения
Linux:
```bash
source .venv/bin/activate
```
Windows (командная строка):
```bash
.venv\Scripts\activate.bat
```

## 3. Запуск интерактивного веб-интерфейса в браузере и обработка предоставленных данных.
```bash
streamlit run app.py
```
Браузер автоматически откроет страницу с интерфейсом CodeLens. Если этого не произошло, перейдите по адресу, указанному в консоли (обычно http://localhost:8501).
## Оценка качества поиска (Precision@5)

Для проверки точности используется тестовый набор из 15 вопросов (`eval_questions.json`) с эталонными ответами.  
Скрипт `generate_results.py` прогоняет все запросы через поисковую систему и сохраняет результаты в `results.json`.  
Метрика **Precision@5** вычисляется скриптом `score.py` (предоставлен организаторами) – ожидаемое значение ≥ 60%.  

**Как запустить оценку:**
```bash
python generate_results.py
python score.py --predictions results.json --questions eval_questions.json
```

## Дополнительные возможности (бонусы)

В проекте реализованы следующие дополнительные функции, повышающие оценку:

- **Генерация связного ответа через LLM** – в интерфейсе есть чекбокс «Включить генерацию ответа ИИ», при активации которого после поиска отправляется запрос к локальной модели `mistral:7b` через Ollama, и пользователь получает краткое техническое пояснение на русском языке.
- **Гибридный поиск** – комбинация векторного сходства и текстовых бонусов за точные совпадения ключевых слов, что повышает точность и интерпретируемость результатов.
- **Поддержка двух языков запросов** – русский и английский (обеспечивается моделью эмбеддингов).
- **Автоматический запуск Ollama** – при старте приложения проверяется доступность локального сервера Ollama и при необходимости он запускается в фоновом режиме.
